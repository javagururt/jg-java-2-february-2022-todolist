На практике большинство систем построено на основе модели событий.
Происходит какое то событие приложение на него реагирует.
Например: пользователь выбрал пункт меню ввода данных ввел
данные и нажал кнопку сохранить, приложение получает запрос
на сохранение данных и должно на него отреагировать (выполнить его).
Запросы бывают сложные с большим количеством параметров и простые
вообще без параметров. Ответ так же могут содержать или не содержать
возвращаемые значения.

Каждое из событий (запросов) требует своей бизнес логики для обработки.
В свою очередь бизнес логика может быть простой и очень сложной, всё
будет зависеть от типа запроса.

Введение request/response поможет нам чётко определить границы каждого
запроса в приложении, позволит нам разбить большое приложение на
отдельную функциональность и разрабатывать каждую функциональность
не зависимо друг от друга.

Создавая для каждого запроса отдельный класс мы следуем принцыпу SRP,
потому что объеденяем все параметры запроса в одном месте.

Создавая для каждого ответа отдельный класс мы тоже следуем принцыпу SRP,
потому что мы объеденяем все параметры ответа в одном месте.
