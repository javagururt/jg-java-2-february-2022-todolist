Где в системе (на каком уровне) должна быть реализована
логика валидации входных данных?

Это очень важный вопрос. UI получает данные от пользователя
и потом передаёт их бизнес логике на выполнение. Первое, что
приходит в голову это реализовать логику проверки входных
данных прямо в нашем консольном UI. Программа запросила ввести
какие то данные, пользователь вводит их и программа на стороне
UI сразу же проверяет введённые пользователем значения и если
в них обнаруживается ошибка (тип данных не тот, данные в неправилтном
формате или вообще пользователь ввел пустое значение) то UI сразу
требует повторить ввод данных еще раз (и так до тех пор пока
пользователь не введёт корректные значения).

Этот подход не правилен по нескольким причинам. Во первых есть
проверки, которые UI не может выполнить самостоятельно. Например
если есть такое правило, которое гласит, что в систему нельзя
добавлять дубликаты. При реализации этой проверки потребуется
база данных. Обращаться напрямую к базе данных с UI минуя
слой бизнес логики это прямое нарушение архитектуры приложения
и в будущем приведёт к запутанному и сложному коду.

Во вторых именно слой бизнес логики отвечает за реализацию всех
бизнес правил в системе - это его ответственность. Реализация
разных проверок входных данных в разных слоях приложения ведёт
к прямому нарушению принципа SRP (Single Responsibility Principle).
Согласно принципу SRP вся валидация входных данных должна быть
реализована в одном месте. Так как все проверки можно реализовать
только на уровне бизнес логики, поэтому именно слой бизнес логики
это правильное место для реализации функционала валидации входных
данных. Уровню бизнес логики доступна напрямую база данных, бизнес
логика может напрямую обращаться к базе данных без нарушения
архитектурных принципов построения приложения.

Некоторые проверки могут быть продублированы на стороне UI. Например
если поле является обязательным для ввода, то UI проверяет введённое
значение пользователем и если оно оказывается пустым, то просто
не посылает запрос на уровень бизнес логиики, а просит пользователя
указать значение в обязательном поле.
